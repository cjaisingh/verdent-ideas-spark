# Migrations

This project uses Supabase migrations. As of 10 May 2026 there are 86 of them, all named with a UUID suffix (`20260507083412_3fa….sql`). That's the Lovable/Supabase default and it's hard to read as an audit trail. We're not renaming the existing ones — too risky against a live schema — but **new migrations follow the convention below.**

## Naming convention (mandatory for new migrations)

```text
<utc-timestamp>__<verb>_<short_subject>.sql
```

- `<utc-timestamp>` — `YYYYMMDDhhmmss` UTC, generated by Supabase
- `__` — double underscore separator (so `index-migrations.ts` can split safely)
- `<verb>` — one of:
  | Verb | Use |
  |---|---|
  | `add` | new table, column, enum value, FK, RLS policy, function, trigger |
  | `drop` | remove a table/column/policy/function |
  | `alter` | change shape of an existing object (rename column, widen type) |
  | `idx` | add/drop an index |
  | `rls` | RLS-only change (enable, policy rewrite) |
  | `fn` | function or trigger body change with no schema impact |
  | `seed` | seed data (constants, lookup rows) |
  | `data` | one-off data migration (backfill, cleanup) |
- `<short_subject>` — snake_case, ≤ 50 chars

### Examples

```text
20260510141500__add_okr_task_link.sql
20260510142000__idx_okr_node_events_node.sql
20260510143000__rls_lessons_operator_only.sql
20260510144500__data_backfill_phase_id.sql
```

## How to write a good migration

- **Single purpose.** One verb per file. If you're adding a table *and* backfilling data, that's two migrations.
- **Idempotent where possible.** `CREATE TABLE IF NOT EXISTS`, `DROP POLICY IF EXISTS`, `INSERT … ON CONFLICT DO NOTHING`.
- **RLS in the same migration that creates the table.** Never ship a public table without policies.
- **Top-of-file comment.** First line must be a `-- ` comment with a one-sentence summary. The migration index parser uses this:
  ```sql
  -- Add okr_node_id + okr_link_kind to roadmap_tasks for Phase 4 outcome wiring
  alter table public.roadmap_tasks ...
  ```
- **No `ALTER DATABASE postgres`.** Not allowed.
- **No edits to Supabase-reserved schemas** (`auth`, `storage`, `realtime`, `supabase_functions`, `vault`).

## Migration index

`docs/migration-index.md` is auto-generated by `scripts/index-migrations.ts`. Re-run any time:

```bash
bun run scripts/index-migrations.ts
```

It parses the first `-- ` comment from each migration and produces a chronological table. For old UUID-named files without a top-comment, the entry shows `(no summary)` — fill these in opportunistically when you touch related code.

## Squash-review cadence

Once we hit 150 migrations, schedule a squash-review sprint: combine clearly-related sequences (e.g. five iterations on the same table) into one consolidated migration, dropped from history. Do this against a staging copy first.
