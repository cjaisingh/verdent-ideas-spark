# AWIP Core — Local Development

This project is built and deployed via [Lovable](https://lovable.dev), but you can run the frontend locally against the same hosted **Lovable Cloud** backend (Supabase) the preview uses. The edge functions and database stay in the cloud — there is no local Postgres or local edge-runtime to manage.

> If you only need to make changes, use the Lovable editor — it auto-installs, auto-deploys edge functions, and applies migrations on approval. This guide is for cloning the repo and iterating locally in VS Code / Cursor / your editor of choice.

## Prerequisites

- **Node.js 20+** and **bun** (`curl -fsSL https://bun.sh/install | bash`)
- **git** with access to the GitHub repo connected to this Lovable project
- A Lovable account with access to the project (for editing secrets, migrations, and edge functions)

Optional, only if you want to run the Supabase CLI locally for edge-function inspection:

- **Deno 1.45+** — `curl -fsSL https://deno.land/install.sh | sh`
- **Supabase CLI** — `brew install supabase/tap/supabase` (or [other methods](https://supabase.com/docs/guides/cli/getting-started))

## 1. Clone & install

```bash
git clone <your-repo-url> awip-core
cd awip-core
bun install
```

## 2. Environment variables

The repo ships a `.env` file managed by Lovable Cloud. **Do not edit it manually** — it's regenerated on every cloud sync. Variables present:

| Var | Used by | Notes |
|---|---|---|
| `VITE_SUPABASE_URL` | frontend + edge function callers | e.g. `https://<project-ref>.supabase.co` |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | frontend (anon key) | safe to expose |
| `VITE_SUPABASE_PROJECT_ID` | tooling | the project ref |

If `.env` is missing after a fresh clone, open the project in Lovable once and the file will be written by the cloud sync.

### Backend secrets (server-side only)

These live in **Lovable Cloud → Connectors → Secrets** and are injected into the `awip-api` edge function at runtime. You don't need them locally to run the frontend.

| Secret | Purpose |
|---|---|
| `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` | edge function → DB writes (bypasses RLS) |
| `SUPABASE_ANON_KEY`, `SUPABASE_PUBLISHABLE_KEY` | server-side auth helpers |
| `SUPABASE_JWKS`, `SUPABASE_SECRET_KEYS` | JWT verification |
| `AWIP_SERVICE_TOKEN` | cross-project auth header (`x-awip-service-token`) used by Discovery AI |
| `LOVABLE_API_KEY` | managed; rotate via Lovable, not the secrets tool |

To add or change a secret, use Lovable Cloud → Connectors. Never commit secret values to git.

## 3. Run the frontend

```bash
bun run dev
```

Vite serves on `http://localhost:5173`. It talks to the **hosted** Supabase project — same database the preview uses, same edge functions. Auth, RLS, and event-stream behavior are identical to production.

To sign in locally, use an account that already has a row in `user_roles` with `operator` (or `admin`). The first signup auto-bootstraps as operator/admin via the `bootstrap_first_operator` trigger; subsequent users start with no role and see auth errors until promoted.

## 4. Database migrations

**Schema changes go through migrations only.** Two paths:

### A. From inside Lovable (recommended)

Ask the agent to make the change. It uses the `migration` tool, you approve the SQL in chat, and the migration runs against the cloud DB. The TypeScript types in `src/integrations/supabase/types.ts` regenerate automatically.

### B. From your local checkout

If you prefer writing SQL directly:

1. `supabase login` (one-time)
2. `supabase link --project-ref <project-ref>` (the value of `VITE_SUPABASE_PROJECT_ID`)
3. Create a new migration: `supabase migration new <name>`
4. Edit the file in `supabase/migrations/`
5. Apply: `supabase db push`

Notes:

- **Never edit `src/integrations/supabase/types.ts` by hand** — it's regenerated by Lovable from the schema.
- **Never run `ALTER DATABASE postgres …`** — not allowed.
- **Never touch the `auth`, `storage`, `realtime`, `supabase_functions`, or `vault` schemas.**
- For data changes (insert/update/delete), don't use migrations — use the Lovable insert tool or write a one-off script.

## 5. Edge functions

The project has one function: `supabase/functions/awip-api/index.ts`.

### Auto-deploy via Lovable (default)

Edits to the function file are picked up on save and deployed automatically. The `verify_jwt` setting is managed; in-code auth (operator JWT or service token) is what enforces access. See [api.md](./api.md#authentication).

### Local invocation (optional)

You can call the **already-deployed** function from your local frontend with no extra setup — that's what `bun run dev` does.

If you want to run the function locally instead:

```bash
supabase functions serve awip-api --no-verify-jwt --env-file ./supabase/.env.local
```

You'll need to provide the secrets listed above in `./supabase/.env.local`. This is rarely worth it — the cloud function is fast to redeploy and shares state with everything else.

### Calling the function

```bash
# Operator JWT (get from supabase.auth.getSession() in the running app)
curl "$VITE_SUPABASE_URL/functions/v1/awip-api/capabilities/demand" \
  -H "Authorization: Bearer $JWT"

# Service token (cross-project)
curl "$VITE_SUPABASE_URL/functions/v1/awip-api/events/recent?limit=10" \
  -H "x-awip-service-token: $AWIP_SERVICE_TOKEN"
```

Full endpoint reference: [api.md](./api.md).

## 6. Running tests

Vitest is configured for the frontend:

```bash
bunx vitest run            # one-shot
bunx vitest                # watch mode
```

Tests live next to components as `*.test.ts(x)` or under `src/test/`. The setup file is `src/test/setup.ts`.

There are no edge-function tests checked in yet. If you add one, place it at `supabase/functions/awip-api/*_test.ts` and run via the Supabase CLI:

```bash
deno test --allow-net --allow-env supabase/functions/awip-api/
```

## 7. Common workflows

### Add a contract endpoint

1. Add the handler to `supabase/functions/awip-api/index.ts`
2. Wire it into the router at the bottom of the file
3. Save → Lovable auto-deploys
4. Test with `curl` or via the operator UI
5. Document it in [api.md](./api.md)

### Add a column / table

1. Ask Lovable to make the migration, or run `supabase migration new …` locally
2. Update RLS policies in the same migration — every table needs them
3. After it lands, the types in `src/integrations/supabase/types.ts` regenerate
4. Update consuming components/edge function code

### Promote a capability

Currently a manual SQL update (no UI yet):

```sql
UPDATE capabilities
SET status = 'available', owning_module = '<module-name>', updated_at = now()
WHERE id = '<capability-id>';
```

Then have the owning module register itself with `POST /capabilities/register` to emit a fresh `capability_events.registered` row.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Frontend logs show 401 from `/awip-api/...` | Logged-in user has no `operator` role | Insert a `user_roles` row, or re-trigger the bootstrap by deleting all rows in `user_roles` and signing up fresh |
| Edge function deploy fails with 500 | Stale `deno.lock` | Remove `supabase/functions/awip-api/deno.lock` and redeploy |
| `supabase db push` fails on a `CHECK` involving `now()` | Postgres requires CHECK constraints to be immutable | Replace with a `BEFORE INSERT/UPDATE` validation trigger |
| Types out of sync after a manual SQL change | `types.ts` only regenerates on Lovable-driven migrations | Run the change as a migration instead |
| `psql` from your shell can't connect | The default `PG*` env vars are only set inside the Lovable sandbox | Use the Supabase Studio SQL editor or the CLI's `supabase db remote` |
